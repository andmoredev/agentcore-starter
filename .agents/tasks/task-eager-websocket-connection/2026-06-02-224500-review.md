# Eager WebSocket connection via app-level context

The WebSocket lifecycle moves out of the Chat component into a dedicated `WebSocketProvider` that sits below `AuthProvider` in the component tree. Once authentication completes, the provider opens a WebSocket connection immediately, before the user navigates to a chat view. This eliminates the connect-on-render latency that previously blocked the first message. The API and service layers make `sessionId` optional so a connection can be established without one; the session is attached per-message instead.

Watch for: Event listeners are wiped on every `disconnect()` call because `WebSocketService.disconnect()` calls `this.listeners.clear()` — this means the context's own `close` listener and all Chat-registered handlers are silently removed after any reconnect cycle (**confirmed**). After a visibility/network reconnect, stream events won't reach Chat.

## High-level view

The provider follows the same context pattern as `AuthContext`: a ref-held service instance, state derived from events, and a hook that throws if used outside the provider tree. Connection is triggered by a `useEffect` watching the auth user, and reconnection is driven by `visibilitychange`/`online` events. The Chat component becomes a pure subscriber that attaches stream handlers and delegates send calls to the context.

The critical flaw is that `disconnect()` clears all registered listeners as a side effect. Because the provider calls `disconnect()` before every `connect()` on the reconnect path, Chat's handlers are removed and never re-registered. This makes reconnection silently broken for message delivery.

Making `sessionId` optional is a minimal, correct change. The backend already falls back to `crypto.randomUUID()` when none is provided, and the frontend now sends sessionId per-message via `sendQuery`.

<summary>Details</summary>

## Listener lifecycle vs. reconnect cycle

`WebSocketService.disconnect()` contains `this.listeners.clear()`. The WebSocketProvider calls `disconnect()` at the top of every `connectWebSocket()` invocation — this is the reconnect path triggered by visibility and network events. Meanwhile, Chat registers its handlers (`stream_event`, `complete`, `error`) in a `useEffect` that depends on `[wsService, handleStreamEvent, handleComplete, handleError]`. The `wsService` reference never changes (it's captured from `useRef` at mount), and the callbacks are memoized with empty deps, so the effect runs exactly once.

After a reconnect: `disconnect()` clears all listeners, `connect()` reopens the socket, but Chat's effect doesn't re-run. The `close` listener registered by the provider's own effect is also gone, meaning `connectionStatus` won't transition to `'disconnected'` on subsequent closes either.

The fix is straightforward: either remove `this.listeners.clear()` from `disconnect()` (only close the socket and reset auth state), or split it into a `close()` that just tears down the socket and a `destroy()` that also clears listeners. The provider would call `close()` on reconnect and `destroy()` only on unmount/logout.

## Presigned URL TTL and idle connections

The backend sets `expiresIn: 300` (5 minutes) on the SigV4 presigned URL. Once the WebSocket is open, the TTL is irrelevant — the connection stays alive. Fresh URLs are fetched on every reconnect, which is correct.

The compound gap: AWS WebSocket APIs typically close idle connections after 10 minutes. The provider is designed to catch this via the `close` event and reconnect on the next visibility change. But the `close` listener is registered once in a standalone effect and gets cleared by the first `disconnect()` call in a reconnect cycle. So after the first reconnect, subsequent server-side closes go unobserved — `connectionStatus` stays `'connected'` while the socket is dead.

## Stale closure in auto-send

The auto-send effect in Chat:

```tsx
useEffect(() => {
  const initialQuery = (location.state as any)?.initialQuery
  if (initialQuery && sessionId && !initialQuerySent.current && connectionStatus === 'connected') {
    initialQuerySent.current = true
    sendMessage(initialQuery)
  }
}, [sessionId, location.state, connectionStatus])
```

`sendMessage` isn't in the dependency array and closes over `isLoading`. If a user navigates to `/chat` with an initial query while a previous send is in-flight (isLoading true), the guard drops the query. The `initialQuerySent` ref prevents retries. This is an edge case but worth noting because the ref makes it non-recoverable.
